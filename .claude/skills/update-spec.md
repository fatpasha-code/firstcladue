# Skill: Update or Create Spec

Использовать когда нужно:
- Создать спеку для новой фичи
- Обновить спеку после изменений в реализации
- Предложить изменение (propose change) и зафиксировать решение

## Типы изменений

1. **Lightweight spec update** — добавить деталь, уточнить описание, обновить поле в схеме
   - Достаточно: обновить TECH_SPEC.md или SPEC_TEMPLATE.md напрямую
   - ADR не нужен

2. **New feature spec** — описать новую фичу перед реализацией
   - Использовать SPEC_TEMPLATE.md как основу
   - Согласовать с пользователем перед реализацией

3. **Architectural change** — меняется принципиальное решение (выбор стека, структура данных, модель данных)
   - Требует ADR (`ADR/YYYYMMDD-description.md`)
   - Требует явного подтверждения пользователя
   - Git commit текущего состояния перед изменением

Не каждое небольшое изменение требует formal proposal или ADR.

## Когда обновлять спеку

- Перед реализацией новой фичи (создать)
- Если изменились требования (с явным подтверждением)
- Если реализация отличалась от спеки — **только после подтверждения пользователем** (см. rules/process.md)

## Процесс создания спеки для новой фичи

1. Взять SPEC_TEMPLATE.md как основу
2. Заполнить все разделы (нет TODO, нет TBD)
3. Сохранить в проект (или добавить раздел в TECH_SPEC.md)
4. Согласовать с пользователем перед реализацией

## Процесс предложения изменения (Propose Change)

Когда нужно изменить что-то уже задокументированное:

```markdown
## Proposed Change: [Что меняем]

**Дата**: YYYY-MM-DD
**Статус**: proposed → applied → archived

**Текущее состояние**:
[что сейчас в спеке/реализации]

**Предлагаемое изменение**:
[что изменить и зачем]

**Причина**:
[почему нужно изменить]

**Влияние**:
- Что меняется в БД (если что-то)
- Что меняется в API (если что-то)
- Что меняется в UI (если что-то)

**Риски**:
[что может пойти не так]
```

Когда изменение применено — обновить спеку и пометить proposal как `applied`.

## Где хранить

- Основная техническая спека: `TECH_SPEC.md` (по модулям)
- Отдельная фича: `specs/FEATURE_NAME.md` или раздел в SPEC_TEMPLATE.md
- Proposals: `ADR/YYYYMMDD-description.md` для важных архитектурных решений

## Проверка согласованности

После обновления спеки убедиться что:
- [ ] TECH_SPEC.md и реализация согласованы
- [ ] CLAUDE.md архитектурная секция актуальна (если изменилась архитектура)
- [ ] ADR создан если принято важное архитектурное решение
