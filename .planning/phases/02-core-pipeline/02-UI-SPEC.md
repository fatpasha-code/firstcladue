### Component behavior contracts

**AnalyzeForm states (4-state pattern):**

1. **Empty (default):** Form visible. Textarea empty. "Анализировать" button disabled. Character counter shows "0 / 50 000".
2. **Ready:** User has typed or pasted text. Button enabled. Character counter updates live.
3. **Loading:** Button disabled, shows staged messages. Form fields disabled. No spinner -- text-based progress only.
4. **Error:** Alert component below form with error message. Form re-enabled for retry.
5. **Success:** Form clears. Result card appears below form with summary + real_status badge + "Открыть запись" link.

## Layout Contract

**Page structure (`/`):**

```
<main> -- centered, max-w-2xl, mx-auto, px-4, py-12
  <h1> -- "Update Tracker", display typography, text-center
  <p> -- user email, text-muted-foreground, text-center

  <Card> -- input form card
    <CardHeader>
      <CardTitle> "Новый анализ"
    </CardHeader>
    <CardContent>
      <form> space-y-4
        <div> -- textarea field
          <Label> "Текст для анализа"
          <Textarea> rows=8, max 50000 chars
          <span> character counter, right-aligned, text-muted-foreground text-xs
        </div>
        <div> -- label field
          <Label> "Метка (необязательно)"
          <Input> text, optional
        </div>
        <Button> "Анализировать" / staged loading text
      </form>
    </CardContent>
  </Card>

  <!-- Shown after successful analysis -->
  <Card> -- result card, gap-6 below input card
    <CardHeader>
      <CardTitle> "Результат анализа"
      <StatusBadge> real_status pill
    </CardHeader>
    <CardContent>
      <p> interpretation.summary
      <a> "Открыть запись" -> /records/[id], text-sm underline
    </CardContent>
  </Card>

  <!-- Sign out form, bottom -->
  <form> signOut action
    <Button variant="outline"> "Sign Out"
  </form>
</main>
```

**Responsive behavior:**
- `max-w-2xl` (672px) at all breakpoints -- single-column centered layout
- `px-4` ensures 16px horizontal padding on mobile
- Textarea `rows=8` provides comfortable input area; grows with content up to max-height of 400px via CSS (`max-h-[400px] overflow-y-auto`)

## Interaction Contract

### Form submission flow

1. User types/pastes text into Textarea
2. Character counter updates on every keystroke (`onChange`)
3. User optionally fills Label field
4. User clicks "Анализировать" (enabled only when text is non-empty and within limit)
5. Button text changes to "Анализируем...", all form fields become `disabled`
6. Staged status messages appear below the button in sequence (D-02):
   - "Сохраняем запись..." (immediately)
   - "Извлекаем данные..." (after record saved, extraction starts)
   - "Интерпретируем..." (after extraction complete, interpretation starts)
   - "Готово" (briefly, before result appears)
7. On success: form clears, result card appears below with fade-in
8. On error: Alert appears below form, fields re-enable, user can retry

### Staged loading implementation

- `useState<string>` for current stage message
- Server Action calls back progress via sequential await steps (save -> extract -> interpret)
- Each step updates state before awaiting the next
- Staged messages are plain `<p>` elements with `text-sm text-muted-foreground animate-pulse`

### Keyboard accessibility

- Tab order: Textarea -> Label input -> Analyze button -> Sign Out button
- Enter in Label input does NOT submit (prevent with `onKeyDown`)
- Cmd/Ctrl+Enter in Textarea submits the form (convenience shortcut)
- All interactive elements have visible focus rings (shadcn default)
