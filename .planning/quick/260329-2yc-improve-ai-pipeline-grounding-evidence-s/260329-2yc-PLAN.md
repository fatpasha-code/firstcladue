Read the current file first.

Add two explicit rules to the extraction system prompt. Place them in the
rules/instructions section, before the output format specification:

**Rule 1 — Qualifier handling:**
```
QUALIFIER RULE: If an item appears to be completed BUT the speaker added a
qualifier indicating it is not final or stable (e.g. "но не финальная",
"пока сырой", "не считаю финальным", "нужно доделать", "временно",
"грубо", "на скорую руку"), classify it as `in_progress`, NOT as `done`.
The qualifier signals ongoing work. Only classify as `done` when the speaker
treats the item as fully resolved with no caveats.
```

**Rule 2 — Full coverage:**
```
COVERAGE RULE: Identify EVERY distinct thematic block in the conversation
(e.g. backend, frontend, analytics, infrastructure, team, finances, product).
For each block, extract at least one item if the block contains actionable or
status information. Do not silently skip topics. If a block was discussed but
produced no extractable items, that is acceptable — but do not omit topics
that have clear done/in_progress/blocker/assignment/deadline information.
```

Do not change the output JSON format of extraction.ts. Do not change the
model selection or API call logic. Only modify the system prompt string.

<task type="auto">
  <name>Task 3: Update interpretation.ts system prompt — grounding-first generation and updated JSON format</name>
  <files>src/lib/ai/interpretation.ts</files>
  <action>
Read the current file first. Also check `.claude/rules/interpretation-prompt-evolution.md`
for process rules before editing.

This change addresses two repeatable failure classes:
1. Unverifiable claims (hidden_blockers/ambiguities as bare assertions)
2. Post-hoc grounding (prohibition against hallucination doesn't prevent it)

**Change 1 — Replace prohibition with generation rule for hidden_blockers and ambiguities:**

Replace any language like "не придумывай скрытые блокеры" / "не выдумывай" with
a positive generation constraint:

```
GROUNDING RULE: For each hidden_blocker and each ambiguity, follow this
two-step process:
1. Find the specific text fragment in the raw conversation that supports this
   claim (quote directly, or paraphrase if quoting is too long). If you cannot
   find a supporting fragment — do NOT include the item.
2. Write the claim AFTER identifying the evidence, not before.
Items without grounding in the raw text are forbidden — including items that
seem plausible or likely given the context.
```

**Change 2 — Update JSON format block to match new schema:**

Update the JSON example/schema description in the prompt to show:

```json
"hidden_blockers": [
  {
    "claim": "...",
    "evidence": {
      "type": "quote",
      "text": "...",
      "speaker": "optional",
      "confidence": "high"
    }
  }
],
"ambiguities": [
  {
    "claim": "...",
    "evidence": {
      "type": "paraphrase",
      "text": "...",
      "confidence": "medium"
    }
  }
],
"real_status_reason": "one sentence explaining why green/yellow/red"
```

**Change 3 — Add real_status_reason instruction:**

```
real_status_reason: one sentence stating the PRIMARY reason for the chosen
status. Reference a specific fact from the conversation (not a generic
statement). Required — do not omit.
```

Do not change the model selection or API call logic. Do not change how
summary or management_view are generated. Only modify the system prompt
string and the JSON format specification within it.
