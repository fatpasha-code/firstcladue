# Benchmark Plan — Interpretation Prompt

## Зачем нужен benchmark set

Без набора тестовых кейсов любое изменение промпта — tuning вслепую.
Одиночный пример не даёт сигнала: может быть лучше на нём и хуже на трёх других.

Benchmark set позволяет:
- Проверять что изменение улучшает класс ошибок, а не один конкретный output
- Фиксировать регрессии до мержа
- Накапливать эталонные outputs для сравнения версий промпта

---

## Типы созвонов в наборе

Набор должен покрывать 6 категорий. Каждая проверяет разный failure mode.

### 1. Спокойный статусный синк
**Что тестируем:** не добавляет ли модель риски которых не было. Fact fidelity.
**Признаки:** всё идёт по плану, нет острых проблем, короткий разговор.
**Ожидаемый real_status:** green.
**Типовая ошибка модели:** придумывает hidden_blockers или ставит yellow без оснований.

### 2. Созвон с явными зависимостями между участниками
**Что тестируем:** правильно ли фиксируются зависимости. Coverage.
**Признаки:** один участник ждёт от другого конкретного артефакта или решения.
**Ожидаемый output:** зависимость явно в hidden_blockers или management_view.
**Типовая ошибка модели:** размывает зависимость или упускает её.

### 3. Хаотичный технический разговор
**Что тестируем:** не поднимает ли модель регистр и не добавляет ли структуру которой не было. Maturity fit.
**Признаки:** прыжки между темами, незакрытые вопросы, неформальный тон.
**Ожидаемый output:** ambiguities отражают реальную незакрытость, не придуманную.
**Типовая ошибка модели:** делает output "слишком структурированным" относительно разговора.

### 4. Релизный созвон с рисками
**Что тестируем:** корректность real_status yellow/red. Status quality.
**Признаки:** явные риски, зависимости от внешних факторов, сроки под давлением.
**Ожидаемый output:** yellow или red с конкретным обоснованием.
**Типовая ошибка модели:** ставит yellow "по умолчанию" или переходит в red без оснований.

### 5. Созвон с малоопределёнными решениями
**Что тестируем:** не придумывает ли модель решения которые не были приняты. Fact fidelity.
**Признаки:** команда обсуждала варианты но не выбрала, вопросы остались открытыми.
**Ожидаемый output:** ambiguities содержат реальные открытые вопросы; решения не выдуманы.
**Типовая ошибка модели:** суммирует дискуссию как принятое решение.

### 6. Разговор где модель склонна переинтерпретировать
**Что тестируем:** enterprise context injection, vocabulary elevation.
**Признаки:** простой неформальный разговор про конкретную задачу.
**Ожидаемый output:** язык output близок к языку разговора, нет "взрослых" терминов.
**Типовая ошибка модели:** добавляет "мониторинг", "спецификации", "масштабирование" там где их не было.

## Структура одного кейса

```
benchmarks/interpretation/cases/{id}/
  input_raw.txt          # исходный текст разговора
  input_extracted.json   # extractedData (можно сгенерировать через runExtraction)
  expected_notes.md      # что должен поймать / не поймать output (не жёсткий эталон)
  category.txt           # номер категории (1–6)
  outputs/               # сохранённые реальные outputs разных версий промпта
    v1.json
    v2.json
    ...
```

`expected_notes.md` — не жёсткий golden output, а набор проверочных утверждений:
- "должен содержать зависимость X"
- "не должен придумывать Y"
- "real_status должен быть yellow потому что..."

---

## Как сравнивать версии промпта

1. Запустить обе версии на всех кейсах benchmark set
2. Оценить каждый output по rubric (`docs/interpretation-eval-rubric.md`)
3. Сравнить средние оценки по каждому критерию
4. Новая версия принимается если:
   - средний балл по целевому критерию вырос
   - нет регрессии (падения >1 балла) по другим критериям
   - на кейсе который был причиной изменения — явное улучшение

---

## Когда правка промпта считается принятой

**Принята:**
- Улучшение подтверждено минимум на 2 кейсах разных категорий
- Нет регрессии на остальных кейсах набора
- Описан класс ошибок который исправлен

**Не принята / требует пересмотра:**
- Улучшение только на одном кейсе (возможно — подгонка)
- Регрессия на кейсах из других категорий
- Нет объяснения какой класс ошибок исправлен

**Если benchmark set пуст:**
- Tuning без benchmark — риск не верифицирован
- Нужно явно это указать при мерже: "tuning without benchmark — не проверено на других кейсах"

---

## Минимальный стартовый набор

Для старта достаточно 3–4 кейсов из разных категорий.
Полный набор из 6 категорий — цель после первых реальных использований инструмента.
